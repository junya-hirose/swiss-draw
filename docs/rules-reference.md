# 競技ルール実装リファレンス

Swiss Draw をデュエル・マスターズ競技イベントの運営に使えるサイトへ育てるための、実装向けルール整理です。

一次資料:

- `dm_competition_rule_20250501.pdf`
- ユーザー提供のPDF本文コピー: `デュエル・マスターズ 競技イベント運営ルール 2025年5月1日発効`

この文書は公式文書の全文転載ではなく、実装に必要な要件へ落とした作業用リファレンスです。細かな裁定例や例外を実装するときは、必ず一次資料本文も確認してください。

## 実装方針

- 大会進行、順位、タイブレーカー、ペナルティは公式ルールを優先する。
- 主催者方針やイベント固有のファクトシートがある場合は、それを設定で上書きできる余地を残す。
- スタッフが大会中に修正・裁定・復旧できることを優先する。
- 参加者画面は確認と報告に集中し、裁定判断は運営側に集約する。
- ルール計算は UI から分離し、テスト可能な純粋関数にする。

## 公式ルールから確認済みの要件

### マッチ構成

実装要件:

- マッチは、必要数のゲームに勝つまで続く一連のゲームとして扱う。
- 通常は1ゲーム先取だが、イベント主催者が必要ゲーム勝利数を変更できる。
- 引き分けゲームは、必要勝利数の計算には入れない。
- ラウンド終了時にどちらも必要勝利数に届いていない場合、ゲーム勝利数が多いプレイヤーをマッチ勝者にする。
- ラウンド終了時点でゲーム勝利数が同じ場合、そのマッチは双方負けとして扱う。
- タカラトミーへ報告されるのはゲーム単位ではなくマッチ結果。

実装メモ:

- 現在の `winnerId: null` は「未報告」「引き分け」「双方負け」を区別できないため、`MatchResult` を導入する。
- 双方負けは standings に両者 loss を加算する専用結果として扱う。
- BO設定は `bestOf` だけでなく `requiredGameWins` として扱うと運用しやすい。

### 時間切れ処理

実装要件:

- マッチ終了時刻に勝者が未決定の場合、現在のターンを終えた後、合計3ターンを追加する。
- 追加ターン終了時にゲームが終わっていなければ、そのゲームは双方負け。
- ジャッジが延長時間を与えた場合、時間切れ処理は延長分が終わるまで始めない。
- シングルエリミネーションでは必ず勝者を決める。ゲーム勝利数が同じ場合は、現在ゲームのシールド数、さらに同数なら予選上位者を使う。

実装メモ:

- スイスラウンドでは「時間切れ処理中」「追加ターン数」「双方負け」の入力UIが必要。
- 決勝ラウンド対応を入れる場合は、スイスとシングルエリミネーションで結果確定ロジックを分ける。

### 時間延長

実装要件:

- ジャッジがラウンドクロック進行中に1分以上マッチを中断させた場合、そのマッチに延長を与える。
- デッキチェックによる中断は、チェック時間に加えて3分の延長を与える。
- 遅いプレイのペナルティでは、時間延長ではなく追加ターンが与えられる場合がある。

実装メモ:

- 延長理由を `judge-ruling` / `deck-check` / `slow-play-extra-turn` / `other` のように分類する。
- 延長秒数だけでなく、追加ターン数も記録できるモデルにする。

### イベントからの退出

実装要件:

- プレイヤーはいつでもイベントから退出できる。
- 第1ラウンド開始前に退出したプレイヤーは、イベント参加者として扱わず、最終順位にも含めない。
- 次ラウンドの組み合わせ生成前に退出を伝えたプレイヤーは、次ラウンド以降のペアリング対象から外す。
- 組み合わせ生成開始後に退出を伝えたプレイヤーは、そのラウンドは組み合わせられる。
- 両者がマッチに現れなかった場合、スコアキーパーへ報告がなければ両者退出扱い。
- 謝礼と引き換えの退出は買収行為として扱う。

実装メモ:

- `PlayerStatus` は `active` / `dropped` / `disqualified` / `removed-before-round1` を分ける。
- Drop操作には「次ラウンドから」「現在ラウンドも不戦敗/不戦勝処理」の選択が必要。
- トップカット後の退出は、代替進出なし・対戦相手不戦勝として扱うため、将来の決勝ラウンド機能で別途扱う。

### 制限時間

確認済みの推奨時間:

| 対象 | 推奨時間 |
| --- | ---: |
| 通常イベント | 15-20分 |
| シングルエリミネーション準々決勝/準決勝 | 50分 |
| シングルエリミネーション決勝 | 無制限 |
| トッキュー8 デッキ登録 | 20分 |
| ドラフト デッキ登録 | 20分 |
| シールド パック開封/カードチェック | 20分 |
| シールド デッキ登録 | 40分 |

実装メモ:

- 現在のデフォルト20分は通常イベントの推奨範囲内。
- ヘッドジャッジが制限時間の最終権限者なので、任意分数を設定可能にする。
- ファクトシート上書きのため、プリセットだけでなく自由入力を維持する。

| 8 | 3 |
| 9-16 | 4 |
| 17-32 | 5 |
| 33-64 | 6 |
| 65-128 | 7 |

## 順位とタイブレーカー

### 公式タイブレーカー順

確認済みの順位決定順:

1. マッチポイント
2. 対戦相手・マッチ・ウィン・パーセンテージ
3. 勝利対戦相手の合計得点
4. 対戦相手がたたかった、相手の対戦相手・マッチ・ウィン・パーセンテージ

実装メモ:

- 現在の勝数優先ソートは公式順と異なる。
- `buildStandings` はマッチポイントを主キーにし、暫定で OMW% を第2キーとして表示・比較する。
- 第3タイブレーカー「勝利対戦相手の合計得点」と第4タイブレーカーの名称・計算は関数名に反映する。
- 第3・第4タイブレーカーは未実装のため、最終提出用順位として扱う前に追加検証する。

### マッチポイント

確認済みルール:

- マッチ勝利: 3点
- マッチ敗北: 0点
- 不戦勝: 3点
- 時間切れでゲーム勝利数が同じ場合: 双方負け
- 時間切れで片方のゲーム勝利数が多い場合: 多い側がマッチ勝利

実装メモ:

- 引き分けマッチポイントは確認できた範囲では定義されていない。スイスラウンドの時間切れ同点は双方負けとして実装する。
- `matchPoints` を `wins * 3` だけでなく結果タイプから計算する。
- `resultType: "double-loss"` は両者に loss、両者0マッチポイントとして扱う。旧保存データは `winnerId` と `playerBId` から resultType を補完する。

### マッチ・ウィン・パーセンテージ

確認済みルール:

- `matchWinPercentage = matchPoints / (playedRounds * 3)`
- 0.33 未満の場合は 0.33 として扱う。
- 途中退出したプレイヤーは、参加したラウンド数を分母に使う。
- 不戦勝は本人のマッチ勝利として扱う。

実装メモ:

- 表示値と比較値は丸め前の数値を保持する。
- 下限補正後の値と、補正前の値をデバッグ表示できると検算しやすい。

### 対戦相手・マッチ・ウィン・パーセンテージ

確認済みルール:

- 対戦相手・マッチ・ウィン・パーセンテージは、対戦した相手のマッチ・ウィン・パーセンテージの平均。
- 不戦勝ラウンドは、この平均計算から除外する。
- 各対戦相手のマッチ・ウィン・パーセンテージにも 0.33 下限補正を適用する。

実装メモ:

- 対戦相手IDリストから計算し、BYEは opponent として保存しない。
- 途中退出者も、参加ラウンド数ベースで MWP を計算して OMWP に含める。
- UIには「計算対象の対戦相手一覧」を出せるようにする。
- 現在の UI は OMW% の数値表示まで。計算対象の相手一覧表示は未実装。

### 未実装・要確認のタイブレーカー

公式タイブレーカー順のうち、以下はまだ実装仕様への分解が必要:

- 勝利対戦相手の合計得点
- 対戦相手がたたかった、相手の対戦相手・マッチ・ウィン・パーセンテージ
- 表示桁数、丸め、同値時の最終ソート

チーム作業では、まず上記2つの計算式を原文から再確認し、テストケースを作ってから実装する。

## ペナルティ

### ペナルティ種別

確認済みの定義:

- 注意: 口頭指示。重大でない誤りや、容易に訂正できる混乱に対して使う。延長時間は与えない。
- 警告: 公式に記録されるペナルティ。解決に時間がかかる不正なプレイに対して使う。1分以上かかった場合は延長時間を与える。
- ゲームの敗北: 該当ゲームを即終了し、違反者はそのゲームに負けた扱いになる。
- マッチの敗北: マッチを適当な時間内に終えられない、またはマッチそのものが成立しない場合の重いペナルティ。
- 失格: イベントの完全性を損なう行為や重大な非紳士的行為に対して与える。

実装メモ:

- `caution` は記録任意に見えるが、運営サイトでは履歴として残す。
- `warning` 以上は公式記録として扱い、イベントレポートに含める。
- `game-loss` と `match-loss` は即時適用と次ラウンド持ち越しの両方を表現できるようにする。
- `disqualification` は現在マッチ敗北、イベント退出、順位表からの除外を同時に行う。

### ペナルティ適用

確認済みルール:

- 注意より厳しいペナルティはイベント報告に添える。
- 遅刻とデッキ/リスト問題を除くゲーム敗北以上は、ヘッドジャッジへの報告およびヘッドジャッジによる適用が推奨される。
- 同時に複数違反がある場合は別々のペナルティとして扱う。
- 同じ原因による複数違反は、最も重いものだけを適用する。
- 同一ラウンドで同じプレイヤーに複数ペナルティを適用できない場合、重いものを先に適用し、残りは次ラウンドへ持ち越す。
- 特定分類に入らない小さな違反は最初は注意で処理でき、繰り返し時は非紳士的行為 軽度として扱う。

実装メモ:

- ペナルティには `appliedRound`, `effectiveRound`, `carryOver`, `requiresHeadJudge`, `reportedToOrganizer` を持たせる。
- UI上は「今回のマッチに適用」「次のゲームに適用」「次のラウンドに持ち越し」を選べるようにする。
- ヘッドジャッジ承認が推奨されるペナルティは、確認ダイアログまたは承認チェックを入れる。

## データモデル改善案

```ts
type PlayerStatus =
  | "active"
  | "dropped"
  | "disqualified"
  | "removed-before-round1";

type MatchResult =
  | "unreported"
  | "player-a-win"
  | "player-b-win"
  | "double-loss"
  | "bye"
  | "forced";

type PenaltyType =
  | "caution"
  | "warning"
  | "game-loss"
  | "match-loss"
  | "disqualification";

type Tiebreakers = {
  matchPoints: number;
  matchWinPercentage: number;
  opponentsMatchWinPercentage: number;
  defeatedOpponentsTotalMatchPoints: number;
  opponentsOpponentsMatchWinPercentage: number;
};
```

追加したい状態:

- `playedRounds`: 不戦勝を含む本人の参加ラウンド数。
- `opponentIds`: BYEを除く対戦相手ID。
- `penalties`: 大会全体のペナルティ履歴。
- `operationLog`: 結果修正、ドロップ、失格、ペアリング再生成の監査ログ。
- `roundConfig`: ラウンドごとの制限時間、決勝/スイス種別、延長設定。

実装要件:

- BO設定に合わせたマッチ結果報告を行える。
- 先攻/後攻を覚えている場合、任意で登録できる。
- Reloadを明確に配置する。
- 現ラウンドの状況を見られる。
- 過去ラウンドの状況を見られる。
- 誰かの勝利やラウンド進行を字幕/ティッカーで表示できる。
- 自分の現ラウンド結果が登録済みの場合、参加者自身がドロップ申請できる。
- ドロップは即確定ではなく、Admin確認を挟める設計にする。

完了条件:

- 参加者がスマホで現在マッチ、対戦相手、テーブル、残り時間をすぐ確認できる。
- 参加者がBOに沿って結果報告できる。
- 登録済み結果後にドロップ申請できる。
- End-User Tester Agentが初見で操作できる。

### ラッキープレゼント抽選

Owner: Admin UX Agent / Domain Logic Agent / End-User Tester Agent

完了条件:

- Adminが対象者一覧を確認して抽選できる。
- 対象外プレイヤーが混ざらない。
- 当選結果が画面とログに残る。

### 予選後の本戦機能

Owner: Domain Logic Agent / Admin UX Agent / QA Agent

目的:

- 予選ラウンド後に上位者を本戦へ進める大会形式に対応する。

実装要件:

- 大会設定で予選ラウンド数をあらかじめ指定できる。
- 予選終了後、上位何名を本戦へ進めるか設定できる。
- 本戦形式を選べる。
  - 本戦もスイスドロー
  - シングルエリミネーション
- 本戦スイスは最後の1人が決定するまで進行できる。
- シングルエリミネーションは3位決定戦あり/なしを設定できる。
- 予選順位と本戦順位を分けて保持する。
- 本戦進出者決定時に、タイブレーカー詳細を確認できる。

完了条件:

- 予選ラウンド数で予選を終了できる。
- 上位N名を本戦へ進出させられる。
- 本戦スイスまたはシングルエリミネーションを開始できる。
- 3位決定戦の有無を設定できる。

### 両敗処理

Owner: Domain Logic Agent / Admin UX Agent

目的:

- 時間切れ等で勝敗が決定していない場合の双方敗北を正しく扱う。

実装要件:

- 両敗は参加者報告ではなく、ジャッジ/Admin側が入力する。
- 両敗は両プレイヤーにマッチ敗北として反映する。
- ゲームスコア、理由、裁定者、メモを記録する。
- タイブレーカー計算に正しく反映する。

完了条件:

- Adminがマッチに両敗を入力できる。
- standingsで両者にlossが加算される。
- 操作ログに両敗理由が残る。

実装メモ:

- Admin のマッチ操作に「両者敗北」を追加し、`resultType: "double-loss"` として保存する。
- 両者敗北は参加者報告ではなく Admin/Judge 操作だけで入力する。

### 大規模大会シミュレーション

Owner: Tournament Simulation Agent / QA Agent

目的:

- 32人、64人、128人、256人の大会が最後までスムーズに進むか検証する。

実装要件:

- テスト用参加者を自動生成できる。
- 同じ seed で同じ結果を再現できる。
- 推奨スイスラウンド数に沿って大会を進められる。
- 全マッチの結果を自動投入できる。
- ラウンド間で保存/再読み込みを挟める。
- standings、pairings、tiebreakers、drop/disqualification、final export を検証できる。
- 大規模テーブル一覧が操作不能にならないか確認できる。

シミュレーション対象:

| 参加人数 | 推奨ラウンド数 | 主な確認点 |
| ---: | ---: | --- |
| 32 | 5 | 基本的なスイス進行、順位、結果入力 |
| 64 | 6 | テーブル一覧の視認性、ラウンド進行 |
| 128 | 7 | パフォーマンス、保存/再読み込み |
| 256 | 8 | 大規模表示、最終結果出力、操作負荷 |

シナリオ:

- 通常結果のみ。
- 一部に両敗を含む。
- 結果登録後のドロップを含む。
- 固定卓プレイヤーを含む。
- 失格者を含む。
- ラッキープレゼント抽選対象の確認を含む。

完了条件:

- 全参加人数で大会を完了できる。
- ペアリング生成が停止しない。
- standings に重複、欠落、不正なBYEが出ない。
- 保存/再読み込み後も大会状態が維持される。
- UIが大規模人数でも運営可能な速度と見やすさを保つ。
- 最終リザルトが出力できる。
